# DM1 all-graphics phase 68 — wire in-game capture as real CMake target

## Problem

The apparent rainbow/static live-viewport issue was made worse by a stale ad-hoc `build/capture_ingame_series` binary. The capture source had already been updated to decode Firestaff's packed framebuffer values with `M11_FB_DECODE_INDEX` / `M11_FB_DECODE_LEVEL`, but the binary was not a tracked CMake target, so normal builds did not rebuild it.

Result: fresh-looking screenshot directories were actually generated by an older capture executable using the obsolete palette/index path.

## Change

- Added `capture_ingame_series` to `CMakeLists.txt` as a first-class executable target.
- Added the missing `render_sdl_m11.h` include to `verification-screens/capture_firestaff_ingame_series.c`, so the framebuffer decode macros are declared when built cleanly through CMake.

## Visual verification

Fresh correctly rebuilt capture:

- `verification-m11/cmake-capture-fixed-20260425-151711/01_ingame_start_latest.png`
- `verification-m11/cmake-capture-fixed-20260425-151711/05_ingame_after_cast_latest.png`

Image review: major rainbow/static corruption is gone in the correctly rebuilt capture. The spell/menu frame is presentable; remaining issues are normal dark/dithered viewport roughness and edge/background cleanup rather than the previous false rainbow failure.
