# Firestaff V2.2 Enhanced 4K/1080p pipeline scaffold

## Scope

V2.2 Enhanced is a new asset-production lane for improved Firestaff V2 art. This scaffold defines the contract for future tooling and manifests only. It does not generate art, does not copy original DM1 bytes, and does not alter V1, M10, or runtime rendering code.

## Production contract

- Each logical asset starts from the measured DM1 logical source size.
- The V2.2 Enhanced master is exactly **10x** the logical source size. This is the 4K-class production master, even when the exact pixel dimensions are asset-specific rather than a full 3840x2160 canvas.
- The 1080p shipping derivative is produced only by exact **50% downscale** from the approved 10x master.
- The derivative is therefore **5x** the logical source size.
- Masters and derivatives are handed to the checksum store before any manifest references them.
- Provenance metadata is required for every asset before ingestion.

## Separation from V2.1 upscaled originals

V2.1 remains the upscale/original-reference lane. V2.2 Enhanced must stay separate:

- V2.2 paths live under `assets-v2/v2.2-enhanced/`.
- V2.2 recipe entries use `assetLineage: "v2.2-enhanced"`.
- V2.2 provenance must not use `sourceKind: "v2.1-upscaled-original"` or `sourceKind: "dm1-original-extract"`.
- V2.2 outputs must not share V2.1 Wave 1 directories such as `assets-v2/ui/wave1/`.
- If an artist uses a V2.1 image as a visual reference, that belongs in a prose provenance note; the V2.1 file is not an input blob for the V2.2 store object.

## Required provenance fields

Each asset recipe entry carries a `provenance` object:

- `sourceKind`: one of the future approved V2.2 production origins, for example `enhanced-clean-room-master`, `licensed-paintover`, or `manual-redraw`.
- `sourceNotes`: short human-readable source and review context.
- `operator`: person or automation lane that prepared the master.
- `toolchain`: toolchain summary sufficient to reproduce the derivative step.
- `licenseNotes`: confirmation that no unlicensed original asset bytes are redistributed.

The dry-run tool validates that the fields exist and are non-empty. It intentionally does not bless art quality.

## Checksum-store handoff

The scaffold uses the existing content-addressed store root:

```text
assets-v2/store/sha256/xx/<sha256>.<ext>
```

A production ingest pass should:

1. Validate the recipe with `tools/v2_upscale_dry_run.py`.
2. Render or provide the V2.2 10x master outside this scaffold branch.
3. Produce the 1080p derivative by exact 50% downscale from that approved master.
4. Ingest both files with `tools/v2_asset_store.py ingest` or the successor batch tool.
5. Record the emitted store paths and SHA-256 values in the final V2.2 manifest/provenance records.
6. Keep manifests repository-relative; never store workstation paths.

## Dry-run commands

```bash
python3 tools/v2_upscale_dry_run.py --emit-template --pipeline-version v2.2-enhanced-4k1080
python3 tools/v2_upscale_dry_run.py --recipe docs/v2/v2.2-enhanced-recipe.example.json
```

Both commands are safe on a clean checkout. They print JSON only and do not create assets.

## Non-goals

- No real art assets in this branch.
- No renderer changes.
- No V1/M10 behavior changes.
- No manifest activation of V2.2 assets.
- No quality claim until visual review approves both the 10x master and the 1080p derivative.
